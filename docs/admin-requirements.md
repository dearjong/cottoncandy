## 1. 문서 목적

사용자 요구사항 문서에는 **운영자(Platform Admin)** 항목이 `ADM-001`~`ADM-004` 네 줄로만 있고, **요구사항명·세부 승인 기준이 비어 있음**.  
본 문서는 다음을 한곳에 묶는다.

1. 기존 **ADM-xxx**를 실무 수준으로 풀어 쓴 요구사항  
2. 사용자 측 요구(`CPJ`, `PJM`, `POF` 등)와 **운영자가 대응해야 하는 업무**의 대응 관계  
3. **현재 cottoncandy 구현**과의 매핑(화면·라우트)

> 구현은 대부분 **목업·로컬 상태**이며, API·권한·감사 로그 영속화는 추후 과제다.

## 2. 사용자 요구사항과의 추적 (역주)

| 사용자 요구 ID | 사용자 메뉴/기능 | 운영자(Admin) 측 대응 업무 |
| --- | --- | --- |
| **CPJ-020** | 프로젝트 승인신청 | 공개(공고) 등 프로젝트 **승인·반려**, 승인 대기 큐 처리 |
| **POF-003** (기업정보) | 사업자정보 인증 | 기업 **인증 신청 검토·승인/반려** |
| **POF-xxx** (포트폴리오) | 회사소개서·포트폴리오 | 콘텐츠 **검수·노출 관리** (구현: 회사소개서&포트폴리오 화면) |
| **PJM-001~022** | 프로젝트 관리 전반 | 단계별 **모니터링·분쟁·이슈 대응**, 계약/정산/리뷰 운영 뷰 |
| **HDR-003**, 알림 | 알림 | 공지·배너·**알림 정책/설정** (구현: 운영 센터 일부) |
| **GID-004** | 1:1 문의 | CS·**커뮤니케이션 센터**와 연계 가능 |

기존 문서의 **ADM-001~004**는 아래 **§4 표**에서 세분화·정리한다.

## 3. 운영자 업무 목표 (비전에 준하는 한 문장)

**공개·신뢰·안전**: 공개 프로젝트와 기업 인증은 운영자 검증을 거치고, 신고·이슈·중단/취소는 정책에 따라 처리되며, 주요 조치는 추적 가능해야 한다.

### 5.3 핵심 구현 메모 (`project-management` 등)

## 10. 필수 운영·보안 요구(초안)

> 전제: 현재 구현은 대부분 목업/로컬 상태일 수 있으므로, 아래는 “반드시 필요하지만 문서에 아직 약한 축”을 문서상 선반영하는 목적입니다.

### 10.1 RBAC(역할 기반 접근제어) — 역할/권한 매트릭스

역할(예시)

| 역할 | 책임 범위(요약) |
| --- | --- |
| `SUPER_ADMIN` | 최고 권한: 사용자/정책/감사/플랫폼 설정 전반 |
| `ADMIN_OPERATOR` | 승인·인증·운영 처리: 프로젝트/신고/분쟁 등 실무 처리 |
| `OPS_CS` | 커뮤니케이션/문의 답변 중심(열람/응대 제한) |
| `ANALYTICS_READONLY` | 통계/리포트 조회 전용(강제 액션 불가) |

권한(예시 액션)

| 권한 항목 | 설명 | `SUPER_ADMIN` | `ADMIN_OPERATOR` | `OPS_CS` | `ANALYTICS_READONLY` |
| --- | --- | --- | --- | --- | --- |
| `PROJECT_APPROVE_REJECT` | 프로젝트 승인/반려 | O | O | X | X |
| `PROJECT_FORCE_STATUS` | 노출/진행 상태 강제 변경(사유 필수) | O | O | X | X |
| `ACCOUNT_VERIFY_APPROVE` | 사업자/인증 승인·반려 | O | O | X | X |
| `SANCTION_ACTION` | 제재(경고/정지/영구정지 등) | O | O | X | X |
| `CONTENT_PUBLISH_EDIT` | 공지/배너/FAQ/약관 게시·수정 | O | O | X | X |
| `ADMIN_SETTINGS_EDIT` | 관리자/플랫폼 설정 변경 | O | (권한 제한) | X | X |
| `AUDIT_LOG_VIEW` | 감사 로그 열람(필터/내보내기) | O | O | (권한 제한) | X |
| `SENSITIVE_DATA_VIEW` | 민감 필드 열람(사유 필수) | O | O | (권한 제한) | X |

> 표의 “(권한 제한)”은 최소 원칙으로, `OPS_CS`는 민감 데이터·감사 로그·강제 액션에 기본적으로 접근하지 못하도록 설계합니다.

### 10.2 감사 로그(Audit Log) — 필수 기록 대상/보관

기록이 “무조건 필요”한 이벤트(예시)

| 이벤트 카테고리 | 예시 액션 | 목적 |
| --- | --- | --- |
| 프로젝트 검토/운영 | 승인/반려, 중단/취소, 강제 상태 변경 | 운영 판단 추적 |
| 인증/제재 | 사업자 인증 승인·반려, 제재 조치(레벨 변경) | 분쟁/책임 추적 |
| 콘텐츠/정책 | 공지/배너 게시·수정, 약관/정책 문서 반영 | 법적/운영 근거 |
| 보안/열람 | 민감정보 열람 시 “열람 사유” 포함 기록 | 개인정보 보호 대응 |
| 시스템 설정 | 유지보수 모드/자동승인/알림 발송 제어 변경 | 장애·오남용 원인 추적 |

보관/접근 원칙(초안)

| 항목 | 요구 |
| --- | --- |
| 보관 기간 | 최소 12개월(권장 24~60개월, 서비스 성격에 따라 조정) |
| 삭제/수정 | 삭제 불가, 관리자라 해도 수정/삭제는 불가(예외는 감사 로그에 별도 기록) |
| 열람 권한 | `AUDIT_LOG_VIEW` 권한을 가진 역할만 조회 가능 |

### 10.3 민감 정보 열람 통제(열람 사유 + 마스킹)

민감 정보 열람이 필요한 상황(예시)

- 메시지/프로필/사업자번호 등 개인정보 또는 민감 식별자가 포함된 데이터
- 제안서/계약/정산 등 법적·재무 관련 문서의 일부

필수 규칙(초안)

| 규칙 | 내용 |
| --- | --- |
| 열람 사유 | 민감 정보 열람 시 운영자가 사유를 입력해야 함(필드 필수) |
| 열람 마스킹 | 기본 표시에서 일부 값만 마스킹하고(예: `***-****-1234`), 원문은 “권한+사유+감사 기록”이 있을 때만 노출 |
| 재열람 | 사유가 동일해도 매 열람 이벤트마다 감사 로그에 기록 |

> 문서 매핑: `ADM-021`(신고 관리), `ADM-022`(운영 활동 로그), `ADM-020`(보안자료/접근 기록)에서 역할/로그/마스킹 정책을 이 섹션으로 일원화합니다.

### 10.4 제재(행정 처분) 레벨 정의

제재 레벨(초안)

| 레벨 | 의미 | 제한 범위(예시) | 복구/종료 |
| --- | --- | --- | --- |
| `WARN` | 경고 | 특정 기능 제한 없음, 정책 위반 기록만 남김 | 자동 또는 수동 해제 |
| `SUSPEND` | 일시 정지 | 프로젝트 등록/참여/발송 일부 제한 | 기간 종료 후 자동 복구 |
| `TERMINATE` | 영구 정지 | 계정 기능 전반 제한 | 복구 불가(또는 별도 승인 프로세스) |
